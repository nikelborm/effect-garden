Findings from the 2026-06-25 session on the next-midi-demo playback scheduler. Companion to [[project_next_midi_demo_scheduling]]. The split below is deliberate: the user asked to separate what they stated explicitly from what I inferred.

## VERIFIED — stated explicitly by the user

- Patterns are little loops split into equal ~1-second chunks ("slots"/ticks); transitions at chunk boundaries sound good. Each tick has a tail "buffer / bad zone" — scheduling a fade-out there collides with the in-progress fade and causes audio artifacts, so you postpone to the next slot.
- The transition queue holds at most 3 elements; a cleanuper removes the oldest; the newest is scheduled on user input — but NOT when the user presses an already-playing element.
- MVP = handle all button transitions EXCEPT the rare ones (e.g. pattern→pattern→pattern, the full-queue case).
- "Let it crash" if an asset isn't downloaded yet; assume all assets are downloaded. The real (deferred) bug is that such signals get produced at all — the disabled-button state used to gate signal creation and was lost in the refactor.
- The refactor regressed a previously-working app; much working logic is commented out; disabled-button rendering is gone. It's "~10% working": can start a sound, cannot stop or properly switch.
- While a pattern is playing, a different accord/strength should switch the loop — NOT immediately, but at end of tick, on the closest tick only if not in the bad zone near the end of the current tick.
- ONLY trusted spec: `advancePlayback/advancePlayingPattern.spec.txt` (user considers it finished / an example). `advancePatternPatternTransition.spec.txt` is an abandoned half-written draft — NOT trusted.
- Code should mirror the specs. The zone test `current_time in queue[0].good_zone` should be a boolean function; `calcTimingsMath` wrongly "ate" that logic by auto-bumping internally.
- Git history is not trustable — don't mine it.
- Many transition states and advancers don't exist as files yet (combinatorial explosion).
- THE FEAR: the app was built assuming a single discrete grid of equal tick intervals with all assets phase-locked. Slow strums can be interrupted mid-play by a new slow strum, which shifts ("sends to hell") the whole grid. This realignment is unsolved and the user's biggest fear. In an ideal grid, good/bad zones derive from `startedAtSecond` alone; slow strums break that confidence. The user for now pretends slow strums don't exist to ease immediate complexity that's being considered.
- Data-class design principle: types in `types/` should store RAW timing VALUES captured at creation (enough to recreate any conclusion), NOT conclusions. `withAdditionalPostpone` is explicitly called bad design.
- `fadeoutStartsAtSecond`/`fadeoutEndsAtSecond` are NOT dead — they were commented out "of despair" to silence type errors and WILL be revived. Cleanupers must know when it's safe to GC and whether an asset finished playing, so finishing timestamps matter. (This corrects my earlier wrong "dead fields" claim.)
- Tests aren't feasible yet: schedulers create opaque WebAudio nodes you can't query (scheduled / in-progress / done), so they're untestable. Make the data derivable/testable first; defer tests.
- PatternSilence advancer behavior (user-specified): the only currently-known way to enter loop→silence is pressing the same pattern to disable it. On PatternSilence + a different pattern: green zone → requalify to PatternPattern; red zone → a NEW `PatternSilencePattern` state. Same pattern + green zone → cancel the fade-out → PlayingPattern. Accord → it's a slow-strum request → die for now (deferred). Strength → just switch the setting, no new playback.
- Kill the registries: they mirror the scheduler loop and drift — truth living outside that can become outdated/lying. Selection must be handled inside the scheduling loop. Silence AND to-silence transitions carry the base accord/strength; all other states already carry it via their asset(s).
- `allAccords`/`allPatterns`/`allStrengths` → standalone sync-layer tags (3 new tags replacing the registry tags). `CurrentlySelectedAssetState` is an old-codebase artifact → delete. The selected-accord/pattern/strength change streams should be inferred from the audio-playback state; for now stub them with dumb values, real inference later. Chose "full teardown now."

## UPDATE (same session, later): #4 implemented end-to-end

- `advancePatternPatternTransition` (#4) is now fully implemented AND dispatcher-wired — both halves the user described:
  - **Green zone** (`now <= latest.fadeInStartsAtSecond - schedulingSafeBufferInSeconds`): the incoming loop hasn't committed, so we cancel/reschedule it — *replace the 2nd queue element*. Inputs: re-press dest accord/strength → no-op; press dest pattern again → deselect → `PatternSilenceTransition`; press the outgoing loop → cancel switch → `PlayingPattern`; anything else → retarget (re-anchor current's fade-out on a fresh slot + fade in the new loop).
  - **Red zone**: the crossfade has begun, `latest` is becoming audible — we can't take it back, so *append a 3rd*: promote `latest` to a fading-out element (own fade-out + cleanup fiber), leave `current` untouched. Retarget → `PatternPatternPatternTransition`; deselect → `PatternPatternSilenceTransition`. (A 4th input on those states = the excluded rare case → default `dieMessage`.)
- Type change: added `ScheduledPatternTransitionQueueElement` (extends the bare queue element) carrying raw `fadeInStartsAtSecond`/`fadeInEndsAtSecond` — the incoming element's go-live window. It is the LAST slot of every transition-queue tuple (the only not-yet-cleaned element). This was the "enrich the queue element with its go-live second" prerequisite.
- **Cleanup-fiber mechanism (non-obvious, verified by reading `makeCleanupFibers.ts`):** a cleanup fiber is just a TIMER. After its delay it opens a latch → runs `SubscriptionRef.updateEffect(stateRef, getNewCleanedUpState)` under the state semaphore. `getNewCleanedUpState` collapses whatever state is CURRENT by one level, gc-ing the OLDEST element. So each element's fiber drives one collapse; with two fibers (e.g. PatternPatternPattern) they fire in slot order (oldest slot first) and collapse the queue down level-by-level. This is why red-zone append works by adding a fiber for `latest` and leaving `current`'s fiber alone. Serialization against `advancePlayback` is via the same stateRef semaphore, so no races / no double-gc.
- `bun run typecheck` + `biome check` green after #4.

## UPDATE 2026-06-26: two fade lengths + PatternSilencePatternTransition (#7)

- **VERIFIED (user):** there are TWO distinct fade lengths. (1) pattern→pattern roll-over = near-instant (`transitionTimeInSeconds`, 1ms) — just declick so a full-volume loop "rolls over" to the next on the tick. (2) fade-to-**silence** (a loop STOPPING) = a real, audible fade; user set `fadeToSilenceTimeInSeconds = 0.3` (300ms) as a **plain value placeholder** (not a multiplier), expected to grow later, possibly multi-tick. Applied to every stop-to-silence path "everywhere now."
- **VERIFIED (user):** "silence" in a state name is SEMANTIC (a loop's stopping trajectory), NOT an audible gap and NOT a queue element. So `PatternSilenceTransition`'s queue is just `[fadingLoop]`, and #7 is `[dyingLoop, incomingLoop]` — the new loop rolls in immediately at the tick, no silence gap.
- Implemented: `zones.ts` `slotEndingAt`/`zoneAt` now take `fadeDurationInSeconds` (default = roll-over) so a longer fade widens its own bad zone (must begin earlier). Crossfade callers use the default; stop-to-silence callers pass `fadeToSilenceTimeInSeconds`.
- Implemented #7: new state `PatternSilencePatternTransition` (`[WithScheduledCleanup dying, Scheduled incoming]`, same shape as PatternPattern). Created in `advancePatternSilenceTransition`'s RED zone (green still requalifies to PatternPattern — and lands cleanly under 300ms since the new loop snaps in at the same boundary the stop-fade ends). Cleanuper collapses it (gc dying → PlayingPattern{incoming}). Advancer `advancePatternSilencePatternTransition` (dispatcher-wired) handles inputs during it: no-op / deselect→silence (green drop incoming→PatternSilence, red fade-both→PatternPatternSilence) / revive-the-dying-loop (green→PlayingPattern) / retarget (green replace 2nd→PatternSilencePattern, red append 3rd→PatternPatternPattern).
- **Robustness note (verified by reasoning, same as #4):** cleanup is position-based — `getNewCleanedUpState` always gc's the OLDEST queue element — so multi-fiber states collapse correctly regardless of which fiber fires first; the Scheduled tail has no fiber and never triggers a collapse.
- `bun run typecheck` + `biome check` green on all new/edited files (pre-existing warnings remain only in commented-out slow-strum stubs).

## UPDATE 2026-06-26: queue elements refactored into a per-loop lifecycle state machine

- **VERIFIED (user decisions):** the queue *elements* (not the high-level states) became classes whose methods return Effects, encapsulate the next sub-state, perform the real audio/scheduler/cleanup side-effects, and enforce invariants (cancelling a fade also un-arms its cleanup fiber). Advancers shrank to: classify the signal → call element methods → assemble returned elements into a freshly-constructed high-level state. High-level state classes were NOT dissolved. User chose: **5 separate element classes**; **store the grid origin (`playbackStartedAtSecond`) on each element** so methods compute their own Slot via `zoneAt`/`chosenSlot`; **everything pulled from Effect context** (no deps args) — incl. a new `CleanupFiberMaker` Context.Tag replacing the `makeCleanupFibers` closure that was threaded as `deps` (the `deps.ts`/`AdvancePlaybackDeps` plumbing is DELETED; provided per-run in `AppPlaybackStateService` where the state ref exists).
- The 5 classes in `types/loopElements.ts`: `PlayingLoopPlayback` (`.beginShortFadeoutBeforeAnotherLoop`→ShortFadeout, `.beginLongFadeoutToSilence`→LastPlayLongFadeout); `IncomingLoopFadingIn` (static `.schedule`, `.drop`→Disposed, `.promoteToFadingOut`→Short, `.promoteToFadeToSilence`→LastPlay, `.becomeLive`→Playing — pure); `LoopPlaybackScheduledWithShortFadeoutBeforeAnotherLoop` and `LoopPlaybackAtItsLastPlayWithScheduledLongFadeout` (both: `.cancelFadeoutAndRestore`→Playing, `.reanchorFadeoutOnto`→Short, `.dispose`→Disposed); `DisposedLoopPlayback` (terminal sentinel, never stored in a queue). Shared side-effect bodies live in free helpers (`scheduleRolloverFadeout`/`scheduleSilenceFadeout`/`reviveToPlaying`/`reanchorToRollover`/`disposeOf`) so the two fade-out classes don't duplicate logic. `getNewCleanedUpState` now routes through `.dispose()`/`.becomeLive()`.
- **Design reality forced a union:** the two fade-out classes are reused interchangeably across transition tuples (a long silence-fade loop can sit in a `PatternPatternTransition` slot via the silence→green path; a roll-over can end up heading to silence), so the queue slots hold `FadingOutLoopPlayback = Short | LastPlayLongFadeout` and BOTH classes carry the same method set. The *incoming* loop is the only element distinguished by its own type. So the user's "successors differ" rationale for 5-separate didn't fully hold; kept separate anyway per their call.
- **TWO FLAGGED TRADEOFFS (my defaults, preserve-behavior):** (1) **independent `now` reads** — each method reads `currentTime` itself, so a crossfade pair (loop fading out + replacement fading in) reads `now` twice; relies on WebAudio `currentTime` being render-quantum-quantised + the 30ms safety buffer to land on the same Slot. If crossfades audibly drift, thread one shared `now`. Documented in the loopElements.ts header. (2) **short-vs-long fade-to-silence inconsistency PRESERVED** — `advancePatternPatternTransition` red-deselect promotes the incoming with a SHORT roll-over fade (`promoteToFadingOut`) even though it heads to silence, while `advancePatternSilencePatternTransition` red-deselect uses the LONG fade (`promoteToFadeToSilence`). This mirrors the original code exactly; arguably both should be long. Left as a known latent bug, not silently "fixed."
